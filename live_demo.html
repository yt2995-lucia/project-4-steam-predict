How we make this prediction trustworthy

Four design decisions that turn an attractive number into a defensible one.

① No data leakage

Pre-release features only

Owners, total_reviews, CCU, recommendations and metacritic are post-release outcomes — they would leak the answer. We explicitly drop every column matching this keyword list before training. EDA Figure 5B (post-release correlations) confirms why.

② Class imbalance handled

73 / 27 split, ROC-AUC primary

The base rate is heavily skewed toward "successful". Reporting accuracy here would be misleading. We use stratified train/test split, class_weight="balanced", and ROC-AUC (not accuracy) as the primary metric. Decision thresholds are placed above the 73% base rate so a "greenlight" verdict actually carries information beyond the prior.

③ Sampling bias acknowledged

Top 1,000 by ownership = our scope

This sample is not "all Steam games" — it is games that already cleared a baseline ownership bar. So the model's natural use case isn't predicting whether a random indie's debut will hit; it's helping a studio that already shipped one game decide whether to fund a sequel / DLC.

④ Latent features as inputs

Cluster + Topic + UMAP

We feed K-Means cluster ID, LDA topic distribution (10 dims), and UMAP coordinates back into the supervised model as engineered features. Unsupervised structure becomes a soft genre prior the classifier can exploit.
